# Introduction
## Contents
## サブネットはどこに構築される?
一つのサブネットは一つのAZに構築される.
つまり, 一つのサブネットが複数のAZにまたがることはない.

## ALBはどこに構築される?
ALBは一つのリージョン内で稼働することができる.

例えば, ap-northeast-1
- リージョン: ap-northeast-1
- サブネット: ap-northeast-1a, ap-northeast-1c

でALBを作成, 指定したとする.
すると, ALBは
- ap-northeast-1aのサブネット内で一つのエンドポイントを作成
- ap-northeast-1cのサブネット内で一つのエンドポイントを作成

する.
つまり, 各AZに一つずつエンドポイントが作成される.

外部からのリクエストは`alb-12345.ap-northeast-1.elb.amazonaws.con`で受け付けるとする.

さて, どのようにこのDNSに対してリクエストが振り分けられるのか.

1. DNSへリクエストが送られる.
2. Route53がリクエストを受け取る.
3. Route53がリクエスト元の地理的・ネットワーク的な条件に基づいて最適なエンドポイント(良い感じのサブネットに属するインスタンス)をクライアントに返す.
4. クライアントはALBから送られたエンドポイント(ap-northeast-1aかap-northeast-1cに属するALBインスタンスのENI)を受け取る.
5. クライアントはエンドポイント(ALBインスタンス)にリクエストを送る.
6. ALBインスタンスはリクエストを受け取り, ターゲットグループにリクエストを振り分ける.

つまり, ALBはRoute53と連携したサービスなのだ.

## ALBはプライベートサブネットに配置できる?
ALBはプライベートサブネットに配置することができる.
しかし, 外部のインターネットから直接アクセスすることができなくなる.

プライベートに設置されたALBはInternal ALBと呼ばれる.
これは主にVPC内のリソースに対してのみアクセスを許可するために使用される.

## private subnet と public subnetって何が違うの?
IGWに接続されているかどうか.
public subnetではインスタンス(ALBとかEC2)がパブリックIP(もしくはEIP)を持つ場合, インターネットから直接アクセス可能.

private subnetはNATインスタンスを経由してからでないとインターネットにアクセスできない. (もしくはVPCエンドポイントを使う)

また, private subnetでインターネットの接続として許されるのはアウトバウンドのみである.

## ALBはどのサブネットで動作する?
VPC内の指定したサブネットでALBは動作する.
つまり, public subnetにALBを配置してしまえば, 残りはprivate subnetsい配置してしまってよい.

DBインスタンスはもちろん, EC2などもprivate subnetに配置することが一般的である.

## どこまでがAZ?
- ap-northeast-1は**リージョン**
- ap-northeast-1aは**AZ**

ここ間違いがち

## DBに初期のテーブルを作成したい.
いくつか方法はあるが, 最も簡単な方法は, RDSインスタンスにアクセスしてローカル環境からSQLを実行することである.

```bash
docker run -it --rm -v .:/app postgres psql -U postgres -h database-1.cymalqtopmzh.ap-northeast-1.rds.amazonaws.com -d myDB -f /app/album_seed.sql
```

ここで必要になるのが,
- `databse-1.cymal...`で指定されるようなDBのエンドポイント
- セキュリティグループによるアクセス制限(許可)

である.
DBのエンドポイントは, RDSのコンソールから確認できる.
もしくは`aws cli`を使っても確認できる.

```bash
aws rds describe-db-instances --query "DBInstances[*].[DBInstanceIdentifier,Endpoint.Address,Endpoint.Port]" --output table
```

DBインスタンスを作成(もしくは変更)した際に,   
* 接続 > 追加設定 > パブリックアクセス可能  
のラジオボタンを有効にしない限り, インターネットからのアクセスはできない.

また, セキュリティグループによるDBへのアクセス制限も必要となる.
* EC2インスタンス > セキュリティグループ > dbのセキュリティグループ > インバウンドルール > ルールの追加 > 全てのトラフィックを許可する

のように設定する必要がある.

ただし, これらの方法は, セキュリティ的にはあまりよろしくない.

別の方法として, 踏み台サーバーを経由してアクセスする方法がある.

1. 踏み台サーバー(EC2)を作成
    - セキュリティグループの設定で, EC2がRDSにアクセスできるようにする.
    - ingressルール{Type: PostgreSQL, Protocol: TCP, Port: 5432, Source: RDSのセキュリティグループ}
    - EC2にSSHで接続するためのキーペアを作成
2. EC2にSSHで接続
    - `ssh -i <秘密鍵.pem> ec2-user@<EC2のパブリックIP>`
    - `sudo yum install postgresql`
3. EC2からRDSに接続/SQLを実行
    - `psql -h <RDSのエンドポイント> -U <ユーザー名> -d <DB名> -f <SQLファイル.sql>`

この方法では, EC2を経由してssh接続するため, セキュリティ的には安全である.

### SSHのトンネリング
なお, このステップを省略した**SSHトンネリング(SSHポートフォワーディング)**を用いた手法もある.
これはどいういうことかというと, 
ローカルPCからEC2にssh接続し, そのEC2からRDSに接続するという手法である.
これでは, 先程と変わらないかと思われるが, 途中のEC2に接続するのではなく, ローカルPCから直接RDSに接続することができる.
つまり,
ローカルPC -ssh-> EC2, EC2 -vpc-> RDS  
ではなく,
ローカルPC -ssh->(EC2)-vpc-> RDS
のように, 一度のssh接続でRDSにアクセスする手法である.

一般に, 暗号化された通信経路(トンネル)を確立して, ネットワークを通じてデータを転送することを**トンネリング**という.

```bash
ssh -i <秘密鍵.pem> -L 5432:<RDSエンドポイント>:5432 ec2-user@<EC2のパブリックIP>
```
そうすると, ローカルで
```bash
psql -h 127.0.0.1 -U <ユーザー名> -d <データベース名> -f <SQLファイル.sql>
```
のようにすればいい.

なお, ここでもEC2やRDSのセキュリティグループの設定が必要である.
また, EC2の秘密鍵の設定`chmod 400 <秘密鍵.pem>`も必要である.

SSHトンネリングにはいくつか種類がある.
1. ローカルポートフォワーディング
  - これは上述で示したもので, ローカルPCからリモートホストに接続する際に使用する.
2. リモートポートフォワーディング
  - ローカルポートフォワーディングとは逆で, リモートホストからローカルPCのリソースに接続する際に使用する.
3. ダイナミックポートフォワーディング
  - 正直よくわからないが, SOCKSプロキシを使って, ローカルPCからリモートホストに接続する際に使用する(らしい...).

通信経路のことをトンネルと呼ぶのはおそらく
- 外部から見えない
- 特定の出入り口がある
- 安全
の特徴を象ったものだろう.

つまり, これはSSHに限ったものではなく, HTTPSやVPNなどでもトンネルを作れる(作っている).

なお, 別の解決方法として**AWS Systems Manager**の**Session Manager**を使う方法もある.

## ECSってパブリックIPを付与しなきゃいけないの?
IGWはVPC内のリソースがインターネットに接続するためのゲートウェイである.
IGW自体にパブリックIPが割り当てられているわけではないが, IGWを使ってVPC内のリソースがインターネットに接続するためには, IGWに対してルートテーブルを設定する必要がある.  
ALB(アプリケーションロードバランサー)はサブネットに付けられたネットワークインターフェース(ENI)を通じて通信を行う. また, 通常はALBのENIにパブリックIPが割り当てられる.  

しかし, ECSタスクにパブリックIPを明示的に割り当てないと, うまく動作しない事例がある.

もっと良い設定をすれば, タスクにパブリックIPを割り当てずに, VPCエンドポイントを使ってECRにアクセスすることもできる.
なお, 今回は~手抜き~ 節約のため, パブリックIPを割り当てる方法を紹介する.

ECSはコンテナをpullするために, AWSサービスの一つであるECRにアクセスする必要がある.  
そう, ECRはインターネットを経由して接続する必要があるのだ.  
ECRはECSからしたら外部サービスであるため, インターネット経由でアクセスする必要がある. つまり, セキュリティグループのアウトバウンドルールが正しく設定されている必要がある.

セキュリティグループはステートフルなので, アウトバウンドルールを設定する際にはインバウンドルールを設定する必要はない.(多分...)

ECRにアクセスするためには, "ecr:GetAuthorizationToken"をはじめとしたいくつかの権限が必要であるが, これらはデフォルトでECSタスク実行ロールに付与されている.

ただし, VPCエンドポイントを使うことで, インターネットを経由せずにECRにアクセスすることもできる.

### VPCエンドポイント
VPCエンドポイントには, 
- ゲートウェイ型: S3, DynamoDB
- インターフェース型: EC2, ECR, RDS, SNS, ...
の2つがある. (本当は3つある)

今回は, ECRにアクセスするために, インターフェース型のVPCエンドポイントについて説明する.
インターフェース型のVPCエンドポイントを作成するとVPCのサブネット内にENIが作られる. このENIを通してVPC内のリソースがAWSサービスにアクセスすることができる.

例えば, VPCエンドポイントのサービス名を
- com.amazonaws.ap-northeast-1.ecr.api
- com.amazonaws.ap-northeast-1.ecr.dkr

のように設定することで, ECRにアクセスすることができる.
なお, DNSホスト名を有効化しておく設定が必要である.
また, ここでもセキュリティグループの設定を忘れてはいけない.

## Go構造体の埋め込み

以下のような構造体があるとする.
```golang
type User struct {
    Name string
    Age int
}

type Config struct {
    User
    Database string
    Password string
}
```
はて, 構造体の埋め込みは内部に埋め込まれている構造体にアクセスしなくても, 内部に埋め込まれている構造体のフィールドにアクセスできる.

つまり, 
```golang
name := Config.User.Name // ok
name := Config.Name // ok
```
のようにアクセスできる.
まるで, 構造体に構造体のフィールドが展開されたような挙動だが, 
埋め込みを展開と勘違いしていると以下のようなコードを書いてしまう.
しかし, 以下は誤りである.

```golang
config := Config{
    Name: "satosi",
    Age: 25,
    Database: "postgres",
    Password: "password",
}
```

以下のように修正すると動作する.
```golang
config := Config{
    User: User{
        Name: "satosi",
        Age: 25,
    },
    Database: "postgres",
    Password: "password",
}
```

結局, 埋め込まれた構造体のフィールドに**アクセスする際**には, 埋め込まれた構造体の名前を省略することができるが, **初期化する際**には埋め込まれた構造体の名前を省略することはできない.

### コンストラクタを使う
また, 構造体の初期化時にデフォルト値を設定したい状況があるが, そのような場合ではコンストラクタを使う.
```golang
func NewConfig() Config {
    return Config{
        User: User{
            Name: "satosi",
            Age: 25,
        },
        Database: "postgres",
        Password: "password",
    }
}
```
これが最もシンプルで一般的な方法である.

### `init`メソッド(カスタム)を作る
インスタンスを生成した後に, デフォルト値を設定するための`init`レシーバーメソッドを作る方法もある.
```golang
type Config struct {
    Host string
    Port int
}

// 初期化メソッド
func (c *Config) InitDefaults() {
    if c.Host == "" {
        c.Host = "localhost"
    }
    if c.Port == 0 {
        c.Port = 5432
    }
}

func main() {
    config := Config{}
    config.InitDefaults() // デフォルト値を適用
    fmt.Println(config.Host) // "localhost"
    fmt.Println(config.Port) // 5432
}
```

### `init`関数を使う
`init`関数を使って, プロセスの初期にインスタンスを生成してしまう方法もある.

```golang
var config Config
func init() {
    config = Config{
        User: User{
            Name: "satosi",
            Age: 25,
        },
        Database: "postgres",
        Password: "password",
    }
}
```

### サードパーティパッケージを使う
また, サードパッケージだが`modifiers`パッケージを使う方法もある.
```golang
import (
    "github.com/go-playground/validator/v10"
    "github.com/go-playground/mold/v4/modifiers"
)

type Config struct {
    Host string `default:"localhost"`
    Port int    `default:"5432"`
}

func main() {
    config := Config{}

    mod := modifiers.New()
    mod.Struct(&config) // デフォルト値を適用
    fmt.Println(config.Host) // "localhost"
    fmt.Println(config.Port) // 5432
}
```

## SQLのプレースホルダーが機能しねぇ..
golangのsqlパッケージを使って, SQLを実行する際に,
```golang
row := db.QueryRow("SELECT * FROM album WHERE id = ?", id)
```
のようにプレースホルダーを使ってSQLを実行すると, エラーが出ることがある.

実は, 使用するドライバによっては, プレースホルダーの記法が異なることがある.
- MySQL: `?`
- PostgreSQL: `$1`, `$2`, ...

このため, ドライバによっては, プレースホルダーの記法を変更する必要がある.

```golang
row := db.QueryRow("SELECT * FROM album WHERE id = $1", id)
```

## パブリックIP・オープンアドレス・グローバルIP・Elastic IP
AWSでは"パブリックIPアドレス"/"オープンアドレス"/"グローバルIPアドレス"などの用語が出てくるが, これらはどう違うのか.
結論, どれも大体同じである.

### パブリックIP
パブリックIPはインターネット上で一意に識別できるIPアドレスであり, インターネットから直接アクセス可能なIPアドレスである....が, 一時的に割り当てられることが多く, インスタンスの再起動などで変わる. 実質的にグローバルIPアドレスであるが, AWSが管理している範囲のものである.

### オープンアドレス
オープンアドレスはアドレスを指すのではなく, インターネットからアクセス可能なアドレスである状態を指す言葉である. 具体的には, ECSなどでタスクが外部アクセス可能であることを示す.

オープンアドレスが有効な場合, そのタスクは**ロードバランサーを介さずに直接インターネットからアクセス可能**となる.
オープンアドレスとは特定のアドレスを指すものでは無く, インターネットからアクセス可能かどうかの状態を示すものであるので, オープンアドレスという言葉自体がグローバルIPと等しくは無い.  
しかし, オープンアドレスが有効な場合, リソースにパブリックIPが割り当てられることが一般的なので, ほとんどグローバルIPと同じ意味で使われることが多い.

## ECSのメモリエラー
ECSを使おうとして, httpdを起動するタスクを作成する.
そしたら, 次のようなエラーが出た.
```plaintext
API の失敗
理由: RESOURCE:MEMORY

    arn:aws:ecs:ap-northeast-1:186578003896:container-instance/a7eac39a0bc44a28aea5f0158fed91ac
```
はて, 何が原因なんだろうか.

一言で言えば, ECStaskの実行必要なメモリが不足しているということだ.
### ECSのタスクについて
ECSのタスクとは_複数_のcontainerをまとめて動作させる単位・機能である.  

タスクは複数のコンテナを管理するための単位なのだ.  
タスクにはコンテナを管理する機能があり,
- リソース共有と割当
    - CPU, メモリなどをコンテナに割り当てる
- ネットワークの共有
    - タスク内のコンテナは同じNIC(ENI)を共有しているため, ローカルIPで通信できる.
- スケーリングとライフサイクルの管理
    - タスクが複数のインスタンスを横断して動作することができる(ECSが管理している).
    - タスク内のコンテナの異常終了に際し, タスク全体を再起動したりすることができる.
- タスク定義による依存関係の管理
    - DBコンテナが起動してからアプリケーションコンテナが起動するなど, 起動順序を管理できる.
- ログの出力
   - CloudWatchを使ってログを出力することができる.
   - ログやエラーハンドリングがタスク単位で行える.

と言った機能がある.

このため, タスクに割り当てられるメモリには各コンテナで使用されるメモリの合計よりも多くのメモリを割り当てる必要があることがわかるだろう.

上のエラーは, このタスクに割り当てられたメモリがタスク内で使用するコンテナのメモリよりも少ないということを示している.  
タスクにメモリが1GiB割り当てられて, そのタスク内で動作するコンテナが1GiBジャスト必要だとしても, タスクはメモリ不足で動作しない.  
もっと言うと, タスク内で使用するコンテナAが1GiB, コンテナBが2GiB必要だとしたら, タスクには3GiBより多くのメモリを割り当てる必要がある.

タスクの設定はタスク定義で行われる.  
このタスク定義にタスク全体でどのくらいのメモリを確保するのか, どんなコンテナを起動するのか, どのくらいのCPUを割り当てるのかなどを記述する. 
タスクは特定の(おそらくEC2)インスタンス上で動作する.
つまり, ノード(おそらくEC2インスタンス)とは異なり, その上で動作するコンテナを管理するより小さい単位がタスクとなる.  

複数のコンテナを使用した例として,
- ウェブサーバー
- データベースコンテナ
- ロギングコンテナ

などをまとめて動作させたいというニーズがある.
マイクロサービスの単位として, 1つの目的を達成するためのコンテナの集合としてタスクを用いた管理が行われる.

## ECS, タスクロールとタスク実行ロールの違い
ECSを使う際に, IAMロールを設定しなければならない.

- ECS サービスロール:  
ECSがクラスタ内のタスクやサービスを管理するために必要な権限を持つ.
ECSのサービス・タスクの起動, 停止, スケーリングに必要な権限を持つ.
なお, コンソール経由だと自動的に付与されるケースもある.

- ECS タスク実行ロール:  
コンテナ内のアプリケーションがECSサービスやAWSリソースにアクセスするために必要な権限を持つ.
ECRからpullしたり, Cloud Watch Logsにログを出力するために必要な権限を持つ.
つまるところ, ECSがタスクを実行するために必要な権限を持つ.

- ECS EC2 インスタンスロール:
これはEC2ベースのECSの場合にのみ必要である.
EC2がタスクの管理やメタデータの取得, インスタンスの登録などを行うために必要な権限を持つ.

ECSタスク実行ロールは必須であるが, その他はコンソールでFargateを選択した場合には自動的に設定されることがある.

ところで, タスク定義には"タスクロール"と"タスク実行ロール"の2つのIAMロールを指定することができる.
これは一体なんなんだろうか. タスク実行ロールは上で述べたが, タスクロールとは一体なんなんだろうか.
以下に対応表を示す.

| 項目 | タスクロール                           | タスク実行ロール                          |
| ---  | ---                                    | ---                                       |
| 概要 | タスクがAWSリソースにアクセスする権限  | ECSがタスクを実行するために必要な権限     |
| 用途 | S3, DynamoDB, SNS, SQSなどへのアクセス | ECRからpull, Cloud Watch Logsへのログ出力 |
| 必須 | 任意                                   | 必須                                      |

つまり, タスクロールはタスクがAWSリソースにアクセスするための権限を持つロールであり, タスク実行ロールはECSがタスクを(セットアップなどで)実行するために必要な権限を持つロールである.

## AWSトラブルシューティング
AWSでトラブルシューティングを行う一般的な方法を紹介する.
### Log
- Cloud Watch Logs:  
ほとんどのAWSサービスはCloud Watch Logsにログを出力することができる.
ログストリームやフィルターなどを活用してログを確認することができる.
- サービス固有のログ:  
RDSなどは独自のログを持っている. クエリログやエンジンログなどを確認することができる.

### メトリクス
- Cloud Watch Metrics:  
どのくらいリソースを使用しているか, パフォーマンスがどのくらいかなどを確認することができる. EC2やRDSなどのサービスはデフォルトでメトリクスを出力している. これを使用することでリソースのどの部分に負荷がかかっているのかや, ボトルネックとなる部分を特定することができる.

### トレーシング
- X-Ray:  
リクエストのフローをトレースすることができる. アプリケーション内のリクエストの依存関係を可視化することができる.

### アクセス権限の確認
- IAMアクセスdenyログ:  
IAMの"アクセス拒否"エラーをログに出力することができる. これらはIAMロールやポリシーの設定ミスを特定するのに役立つ. 
- IAMポリシーシミュレータ:  
指定したポリシーで特定のアクションが許可されているかどうかをシミュレートできる.

### ネットワーク診断
- VPCフローログ:  
ネットワーク通信の問題に対して, 通信状況を確認することができる. 対象にアクセスできない場合や, サブネット, セキュリティグループの設定ミスを特定するのに役立つ.
- セキュリティグループ・ネットワークACL:  
リソース間の通信を制限するセキュリティグループやネットワークACLの設定を見直す.

### イベント通知の確認
- AWS Health Dashboard:  
AWSサービスの稼働状況や計画メンテナンスなどの情報を確認することができる.
- Cloud Watch アラーム:  
メトリクスがある閾値を超えた場合にアラームを発生させることができる.

### コンソールのデバッグ
AWSコンソール, CLIのデバッグ機能を使用する.
